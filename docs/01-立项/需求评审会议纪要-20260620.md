### 二、技术可行性评审（15分钟）

**评审人**：技术负责人（AI助手）  
**评审结论**：**有条件通过**（需修改后重新评审）

#### 🔴 必须修改（P0）

**问题1：技术架构未确定**
- **位置**：第五章 5.1 技术架构
- **问题**：云函数模式 vs 小程序端计算，PRD写的是"[待确认]"
- **技术建议**：
  ```
  建议v1.0采用【小程序端计算】方案
  理由：
  1. 计算引擎已经是JS代码（pension-engine.js），直接引到小程序就能用
  2. 不需要云函数，省去服务器成本和冷启动延迟
  3. 离线也能算（用户可能在社保局排队，没网）
  4. 包体积增加<50KB（引擎代码已经很小）
  ```
- **行动**：需要产品负责人在评审会上确认这个方案

**问题2：微信支付合规性风险**
- **位置**：第四章 4.4 合规要求
- **问题**："工具-计算器"类目能否支持微信支付？
- **风险**：
  ```
  微信规则：工具类目不能做虚拟商品支付（详细报告属于虚拟商品）
  可能被拒原因："类目与功能不符"
  ```
- **解决方案**：
  ```
  方案A：升级类目到"生活服务-社保"（需要《人力资源服务许可证》）
  方案B：详细报告免费，通过广告变现
  方案C：先做免费版，v1.5再接入支付
  ```
- **行动**：需要产品负责人确认选择哪个方案

**问题3：场景2与需求清单矛盾**
- **位置**：第二章 2.1 场景2 vs 2.2 需求清单
- **问题**：场景2描述"离职测算模式"，但REQ-008标记为P2（v1.5）
- **矛盾点**：产品概述里说"核心使用场景是从企业离职算一下"，但v1.0不做这个功能？
- **行动**：需要产品负责人确认"离职测算"是否做到v1.0里

#### 🟡 建议修改（P1）

**问题4：自动定位体验优化**
- **位置**：第三章 3.1 首页功能逻辑
- **问题**：首次进入就调用`wx.getLocation`，会弹授权框，体验不好
- **建议**：
  ```
  改成：默认手动选择省份，在"设置"里再开启定位
  或者：首页加一个"自动定位"开关，用户主动点击才授权
  ```

**问题5：分享图片生成性能风险**
- **位置**：第三章 3.3 结果页
- **问题**：用`canvas`生成分享图片，Android低端机可能卡顿
- **建议**：
  ```
  用微信官方的"小程序码 + 后端合成图片"方案
  或者：用云函数生成分享图（但需要云函数支持）
  ```

**问题6：项目排期偏乐观**
- **位置**：第九章 项目排期
- **问题**：20天（4周）可能不够
- **建议**：
  ```
  前端开发：7天 → 10天（要适配31省的配置）
  联调测试：3天 → 5天（31省都要测）
  审核上线：3-7天 → 7-14天（首次提审可能被拒）
  
  总计：约6周（30个工作日）
  ```

#### 🟢 可选优化（P2）

**问题7：敏感词检查必要性**
- **位置**：第二章 2.2 REQ-010
- **问题**：用户不输入文字（只输入数字），需要`msgSecCheck`吗？
- **建议**：如果v1.0没有用户评论/反馈功能，可以去掉这个需求

**问题8：数据更新机制**
- **位置**：第七章 7.1 数据来源
- **问题**：社平工资每年更新，目前需要手动改配置文件
- **建议**：v1.5可以加一个"在线更新"功能（从小程序后台拉取最新数据）
