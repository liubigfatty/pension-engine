## 1. 项目概述

### 1.1 项目背景

养老金计算平台旨在为用户提供精准的养老金计算服务，覆盖全国31个省、自治区、直辖市的养老保险政策。项目包含：
- **计算引擎**：支持1990-2025年社平工资数据，31省政策规则
- **小程序端**：微信小程序前端（本手册管辖范围）
- **后端服务**：API服务（见后端文档）

### 1.2 项目目标

| 目标类别 | 具体指标 |
|---------|---------|
| **功能目标** | 支持31省养老金计算，准确率>99% |
| **性能目标** | 启动时间<1.5秒，包大小<2MB |
| **质量目标** | 审核通过率>90%，Crash率<0.1% |
| **运营目标** | 7日留存率>25% |

### 1.3 项目范围

**包含**：
- 微信小程序前端开发
- 计算引擎封装（云函数/API调用）
- 微信生态集成（登录、分享、订阅消息）

**不包含**：
- 后端API开发（见后端项目手册）
- 运营推广（见运营手册）

## 4. 阶段一：立项与准备

### 4.1 输入清单

| 输入项 | 提供方 | 格式 | 备注 |
|--------|--------|------|------|
| 商业需求 | 项目发起人 | Word/语雀 | 描述为什么要做这个小程序 |
| 预算批准 | 财务/管理层 | Excel | 开发成本、运营成本 |
| 公司资质 | 行政/法务 | PDF扫描件 | 营业执照、法人身份证 |
| 类目资质 | 业务方 | PDF扫描件 | 如涉及医疗/金融/教育需特殊资质 |

### 4.2 工作流程

#### 步骤1：需求评审会（第1-2天）
**参与人**：项目发起人、PM、技术负责人、UI设计师  
**输入**：商业需求文档  
**输出**：PRD初稿、项目排期草案

**会议纪要模板**：
```markdown
# 需求评审会议纪要

## 会议信息
- 时间：YYYY-MM-DD HH:MM
- 地点：[会议室/腾讯会议]
- 主持：[PM姓名]
- 参会人：[列出所有参会人]

## 需求讨论
1. [需求点1]：结论/待定
2. [需求点2]：结论/待定

## 决策事项
- [决策1]：理由、负责人、deadline
- [决策2]：...

## Action Items
| 任务 | 负责人 | Deadline |
|------|--------|-----------|
| [任务1] | [姓名] | YYYY-MM-DD |
```

#### 步骤2：类目资质审核（第3-5天）
**参与人**：合规专员、PM  
**输入**：小程序类目清单（见附录A）  
**输出**：资质文件清单、补正计划

**操作规范**：
1. 登录微信公众平台→设置→基本设置→服务类目
2. 查看所需资质（系统会自动提示）
3. 列清清单，交给行政办理
4. 资质文件扫描存档（`docs/01-立项/资质文件/`）

#### 步骤3：UI设计（并行，第3-10天）
**参与人**：UI设计师、PM  
**输入**：PRD文档、竞品分析  
**输出**：原型图、UI设计稿

**设计规范**：
- 基准屏幕：750rpx（iPhone 6/7/8）
- 字体：微信系统字体（PingFang SC）
- 色彩：见品牌规范（如有）
- 组件：优先使用微信原生组件

## 5. 阶段二：技术架构与初始化

### 5.1 输入清单

| 输入项 | 提供方 | 格式 |
|--------|--------|------|
| PRD文档 | PM | 语雀链接 |
| UI设计稿 | UI设计师 | Figma链接 |
| 后端接口清单 | 后端负责人 | Apifox项目链接 |

### 5.2 工作流程

#### 步骤1：技术选型决策（第1天）
**参与人**：前端负责人、后端负责人  
**输出**：技术选型ADR（Architecture Decision Record）

**决策模板**（见templates/ADR模板.md）：
```markdown
# ADR-XXX: [决策标题]

## 状态
[已决定/已废弃/待讨论]

## 背景
[为什么要做这个决策？]

## 决策
[我们决定做什么？]

## 理由
[为什么选这个方案？]

## 替代方案
[还有哪些方案？为什么没选？]

## 影响
[这个决策会影响哪些模块？]
```

**常见技术选型决策**：
1. **框架选型**：原生 vs uni-app vs Taro
2. **状态管理**：globalData vs MobX vs Vuex
3. **UI组件库**：原生 vs Vant Weapp vs TDesign
4. **分包策略**：主包包含哪些页面

#### 步骤2：系统架构设计（第2-3天）
**参与人**：前端负责人  
**输出**：系统架构图、目录结构说明

**架构图要素**：
- 前端页面结构（tabBar、页面层级）
- 数据流向（用户输入→计算引擎→结果展示）
- 微信API调用（登录、分享、支付）
- 后端API对接（RESTful接口）

**工具**：Figma/ProcessOn/飞书文档画图

#### 步骤3：项目初始化（第4-5天）
**参与人**：前端负责人  
**输出**：Git仓库、项目脚手架

**操作规范**：
```bash
# 1. 在微信开发者工具创建项目
# 选择：不使用云服务（我们用自己的后端）
# 勾选：使用TypeScript（推荐）

# 2. 初始化Git仓库
cd [项目目录]
git init
git add .
git commit -m "feat: 项目初始化"

# 3. 推送到工蜂（腾讯Git）
# 在工蜂创建仓库 → 获取远程地址
git remote add origin [远程地址]
git push -u origin master

# 4. 创建分支策略
git checkout -b develop
git push -u origin develop
```

**分支策略**：
```
master       # 生产环境，只允许从release合并
└── release  # 预发布环境，从develop合并
    └── develop  # 开发环境，日常开发分支
        └── feature/xxx  # 功能分支，开发完成后合并回develop
```

#### 步骤4：API接口设计（与后端并行，第3-5天）
**参与人**：前端负责人、后端负责人  
**输出**：API接口文档（Apifox）

**接口设计规范**：
- 使用RESTful风格
- 统一响应格式：
  ```json
  {
    "code": 0,
    "message": "success",
    "data": { ... }
  }
  ```
- 错误码统一（见附录B）

## 6. 阶段三：核心开发

### 6.1 输入清单

| 输入项 | 提供方 |
|--------|--------|
| 技术设计文档 | 前端负责人 |
| UI设计稿（标注版） | UI设计师 |
| API接口文档 | 后端负责人 |

### 6.2 工作流程

#### 开发模式：Scrum敏捷开发
**迭代周期**：2周一个Sprint  
**每日站会**：15分钟，更新任务看板

#### 步骤1：Sprint规划会（每个Sprint第1天）
**参与人**：全体开发团队  
**输入**：PRD、优先级排序的需求池  
**输出**：Sprint目标、任务拆解

**操作规范**：
1. 从Tapd需求池选择本Sprint要做的需求
2. 拆解成任务（每个任务不超过8小时）
3. 认领任务，更新看板状态

**看板状态**：
```
待办 │ 进行中 │ 开发完成 │ 测试通过 │ 已发布
```

#### 步骤2：每日开发（Sprint第2-9天）
**编码规范**（见第11节）

**每日工作流**：
```bash
# 1. 拉取最新代码
git checkout develop
git pull origin develop

# 2. 创建功能分支
git checkout -b feature/养老金计算器

# 3. 开发功能
# ... 编码 ...

# 4. 提交代码（规范见11.2节）
git add .
git commit -m "feat(计算器): 添加输入表单组件"

# 5. 推送到远程
git push origin feature/养老金计算器

# 6. 创建Merge Request（工蜂/GitLab）
# 填写：功能说明、测试步骤、关联Tapd任务ID

# 7. Code Review（至少1人Approve才能合并）
# 在工蜂MR页面进行

# 8. 合并到develop分支
# 由MR创建人操作（通过工蜂界面）
```

#### 步骤3：接口联调（贯穿整个开发阶段）
**参与人**：前端开发、后端开发  
**工具**：Apifox

**联调规范**：
1. 后端先定义接口（Apifox）
2. 前端根据接口文档 mock 数据（Apifox自动生成）
3. 后端开发完成后，在Apifox切换为"真实API"
4. 联合测试，记录问题（Tapd Bug单）

**联调记录模板**（存 `docs/04-开发/接口联调记录/`）：
```markdown
# 接口联调记录 - 养老金计算器

## 接口：/api/calculate
- 联调日期：2024-06-20
- 前端：张三
- 后端：李四
- 问题记录：
  - [ ] 参数名错误：前端传 `salary`，后端期望 `avg_salary`（已修复）
  - [x] 响应格式错误：缺少 `province_name` 字段（后端已加）
- 验收结果：通过 ✅
```

#### 步骤4：Sprint评审会（每个Sprint最后1天）
**参与人**：全体 + PM  
**输入**：本Sprint完成的功- **输出**：Demo演示、改进建议

**演示规范**：
1. 在微信开发者工具"真机调试"演示
2. 录制演示视频（存 `docs/04-开发/Sprint演示/`）
3. PM签字确认（邮件或Tapd评论）

#### 步骤5：Sprint回顾会（Sprint评审会后）
**参与人**：全体开发团队  
**输出**：改进措施

**回顾模板**：
```markdown
# Sprint X 回顾会议

## 做得好的
- [点1]
- [点2]

## 需要改进的
- [问题1]：改进措施
- [问题2]：改进措施

## Action Items
| 改进措施 | 负责人 | Deadline |
|---------|--------|-----------|
| [措施1] | [姓名] | YYYY-MM-DD |
```

**参与人**：测试工程师

**冒烟测试清单**：

- [ ] 核心流程能走通（计算器输入→结果）

| 主包大小 | < 2MB | 开发者工具→详情→基本信息 |
| setData耗时 | < 50ms/次 | 开发者工具→调试器→Performance |
| Crash率 | < 0.1% | 微信公众平台→运维中心 |

**性能优化规范**（见第11.3节）

| 审核材料 | PM、合规专员 |

### 8.2 工作流程

#### 步骤1：准备审核材料（提审前2天）
**参与人**：PM、前端负责人  
**输出**：审核材料包

**材料清单**：
```
docs/06-发布/审核材料-v1.0/
├── 测试账号清单.md  # 审核人员登录用
├── 功能页截图/       # 每个主要页面的截图
├── 小程序介绍文案.docx  # 名称、简介、关键词
├── 类目资质文件.pdf  # 最新版
└── 隐私协议链接.txt  # 必须是HTTPS，且能在线访问
```

**测试账号规范**：
- 提供至少2个测试账号（防止一个失效）
- 账号必须有完整数据（能跑通所有流程）
- 在 `docs/06-发布/审核材料/测试账号清单.md` 写明：
  ```
  测试账号1：testuser1 / 密码123456
  测试账号2：testuser2 / 密码123456
  
  特殊说明：
  - 账号1已绑定广东社保，可测试计算功能
  - 账号2未绑定，可测试绑定流程
  ```

#### 步骤2：上传代码（提审前1天）
**参与人**：前端负责人  
**工具**：微信开发者工具

**操作规范**：
1. 在开发者工具点击"上传"
2. 填写版本号（遵循语义化版本，见11.4节）
3. 填写版本描述（要详细，审核人员会看）
   ```
   示例：
   v1.0.0 2024-06-20
   - 新增：养老金计算器功能
   - 新增：微信登录
   - 修复：已知Bug
   ```
4. 上传成功后，登录微信公众平台确认

#### 步骤3：提交审核
**参与人**：PM  
**工具**：微信公众平台

**操作规范**：
1. 登录微信公众平台→开发管理→开发版本
2. 点击"提交审核"
3. 填写：
   - **功能页面**：填写主要页面路径（如 `pages/calculator/calculator`）
   - **测试账号**：复制粘贴测试账号清单
   - **隐私协议**：填写在线链接
4. 上传**审核视频**（如涉及特殊功能，如VR、直播）
5. 提交

**审核周期**：
- 普通类目：1-3天
- 特殊类目（金融/医疗/教育）：3-7天

#### 步骤4：处理审核反馈（审核期间）
**可能结果**：
- ✅ **审核通过**：进入步骤5
- ❌ **审核驳回**：见步骤4.1

**步骤4.1：审核驳回处理**
**参与人**：PM、前端负责人  
**输出**：驳回分析报告、整改证明

**驳回处理规范**：
1. 在微信公众平台查看驳回原因
2. 分析：是Bug还是政策不符？
3. 如果是Bug → 修复 → 重新上传 → 重新提交审核
4. 如果是政策不符 → 修改功能/补充资质 → 重新提交审核
5. 在 `docs/06-发布/审核沟通记录/` 记录每次驳回和整改

**驳回记录模板**：
```markdown
# 审核驳回记录 - v1.0.0

## 驳回日期
2024-06-22

## 驳回原因
[微信官方反馈的原文]

## 原因分析
[为什么会驳回？]

## 整改措施
- [措施1]：负责人、deadline
- [措施2]：...

## 整改证明
[截图/代码commit/资质文件]

## 重新提审日期
2024-06-23
```

#### 步骤5：发布上线
**参与人**：PM  
**工具**：微信公众平台

**操作规范**：
1. 审核通过后，在微信公众平台→开发管理→审核版本
2. 点击"发布"
3. **可选**：灰度发布（先放5%用户，观察24小时）
4. 全量发布

**发布后验证**：
- 用真实用户账号测试（不要用管理员账号）
- 检查微信搜索能找到小程序
- 检查分享功能正常

#### 步骤6：监控与运维（发布后）
**参与人**：前端负责人、后端负责人  
**工具**：微信公众平台→运维中心

**监控指标**：
| 指标 | 告警阈值 | 处理方式 |
|------|---------|---------|
| Crash率 | > 0.1% | 紧急修复，提交热更新 |
| 错误接口 | > 5% | 检查后端服务 |
| 用户反馈 | 负面>10条/天 | 分析问题，计划修复 |

│   └── icons/
└── subpackages/              # 分包（如有）
    └── user-center/

### 11.2 Git Commit规范

**格式**：
```
<type>(<scope>): <subject>

<body>

<footer>
```

**type类型**：
| type | 说明 | 示例 |
|------|------|------|
| `feat` | 新功能 | `feat(计算器): 添加输入表单` |
| `fix` | Bug修复 | `fix(登录): 修复token过期问题` |
| `docs` | 文档修改 | `docs(README): 更新安装步骤` |
| `style` | 代码格式（不影响功能） | `style(首页): 删除多余空行` |
| `refactor` | 重构 | `refactor(请求层): 改为Promise封装` |
| `perf` | 性能优化 | `perf(计算器): 减少setData调用` |
| `test` | 测试相关 | `test(计算器): 添加单元测试` |

**scope（作用域）**：
- 模块名：`计算器、登录、分享`
- 文件名：`request.js、auth.js`

**subject（简短描述）**：
- 不超过50个字符
- 用中文（团队规范）
- 结尾不加句号

**body（正文，可选）**：
- 说明**为什么**要做这个改动
- 说明**怎么**做的（如果是复杂功能）

**footer（尾部，可选）**：
- 关联Tapd任务：`Closes #123` 或 `Related to #456`
-  Breaking Change（不向下兼容）：注明影响范围

**完整示例**：
```bash
git commit -m "feat(计算器): 添加1990-2023年社平工资数据

- 数据源：各省人社厅官方文件
- 影响文件：provinces/*.json
- 关联需求：PRD-2024-06-001

Closes #123"
```

### 11.3 性能优化规范

#### 启动优化
- [ ] 主包大小 < 2MB（开发者工具→详情→基本信息）
- [ ] 非关键资源延迟加载（`wx.nextTick`）
- [ ] 使用分包预加载（`app.json → preloadRule`）

#### 渲染优化
- [ ] `setData` 频率 < 10次/秒
- [ ] `setData` 数据量 < 100KB/次
- [ ] 使用 `pureDataPattern` 标记非渲染数据
- [ ] 长列表使用虚拟列表（`recycle-view` 组件）

#### 图片优化
- [ ] 图片格式：优先WebP，兜底PNG/JPG
- [ ] 图片尺寸：不超过750rpx（小程序屏幕宽度）
- [ ] 图片懒加载：`lazy-load` 属性
- [ ] 图片CDN：使用腾讯云COS，开启图片压缩

#### 网络优化
- [ ] 请求缓存：高频不变数据缓存到本地（`wx.setStorage`）
- [ ] 请求合并：多个接口合并成一个（后端支持）
- [ ] 离线降级：断网时显示缓存数据

### 11.4 版本号规范

**遵循语义化版本 2.0.0**：
```
v[主版本].[次版本].[修订号]
```

**示例**：
| 版本 | 说明 |
|------|------|
| `v0.1.0` | 第一个内测版本 |
| `v1.0.0` | 第一个正式版本 |
| `v1.0.1` | 修复Bug（修订号+1） |
| `v1.1.0` | 新增功能（次版本+1，修订号归0） |
| `v2.0.0` | 架构重构（主版本+1，次版本和修订号归0） |

**Git Tag规范**：
```bash
# 打Tag
git tag -a v1.0.0 -m "第一个正式版本"
git push origin v1.0.0
```

### 附录B：错误码规范

| 错误码 | 含义 | HTTP状态码 |
|--------|------|-----------|
| 0 | 成功 | 200 |
| 40001 | 参数错误 | 400 |
| 40002 | 用户未登录 | 401 |
| 40003 | 权限不足 | 403 |
| 50001 | 服务器内部错误 | 500 |
| 50002 | 数据库连接失败 | 500 |

**前端处理规范**：
```javascript
// utils/request.js
wx.request({
  // ...
  success: (res) => {
    if (res.data.code === 0) {
      resolve(res.data.data);
    } else if (res.data.code === 40002) {
      // 用户未登录，跳转登录页
      wx.navigateTo({ url: '/pages/login/login' });
    } else {
      reject(res.data);
    }
  }
});
```
