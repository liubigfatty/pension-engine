## 二、扩展策略

### 策略：核心引擎不变，仅替换配置数据

```
养老金计算引擎（全国通用）
├── 基础计算逻辑（不变）
├── 省份配置（可替换）
└── 特殊规则（可选配置）
```

**关键优势**:
- 不需要为每个省份修改代码
- 新增省份只需添加配置
- 保持吉林的复杂计算能力作为模板

## 四、每个省份的标准化开发流程

```
1. 政策调研（2h）
   ├── 收集计算规则
   ├── 收集历年基数
   └── 确认特殊政策

2. 配置构建（4h）
   ├── 创建省份 JSON 配置
   ├── 填入基数数据
   ├── 填入过渡系数
   └── 填入特殊规则

3. 数据验证（2h）
   ├── 使用已有案例验证
   ├── 对比官方计算器
   └── 修正数据

4. 小程序更新（2h）
   ├── 更新省份列表
   ├── 更新数据源
   └── 测试功能

5. 发布（1h）
   ├── 代码审核
   ├── 测试
   └── 发布
```

**每省预计工时**: 11h（批量处理可降低）

## 五、批量处理优化

### 批量配置策略

1. **同类型省份批量处理**: 将过渡系数、视同缴费规则相同的省份放在一起处理
2. **数据批量获取**: 一次性从同一来源获取多省数据
3. **配置模板复用**: 创建配置模板，快速生成新省份配置

### 批量处理分组

| 分组 | 省份 | 处理策略 |
|------|------|---------|
| 1.4% 组 | 西藏、青海、新疆、云南、吉林、陕西、宁夏、广西、贵州、四川、重庆、甘肃 | 使用相同模板，仅替换基数数据 |
| 1.3% 组 | 山东、河南、河北、安徽、江苏、江西、湖北、山西、辽宁、黑龙江 | 使用相同模板 |
| 1.2% 组 | 上海、湖南、广东、浙江、福建、海南、内蒙古 | 使用相同模板 |
| 1.0% 组 | 北京、天津 | 特殊处理 |
| 特殊组 | 广东（5 年过渡）、北京（两阶段） | 单独处理 |

**优化效果**: 批量处理可减少 40-50% 工时
